# CommentLab：多 Agent 社交媒体内容发布风险模拟系统选题报告

## 一、项目基本信息

**项目名称：** CommentLab  
**中文名称：** 面向个人创作者的多 Agent 社交媒体内容发布风险模拟系统  
**开发形式：** 两人小组：陈思琪、张若西  
**预期形态：** 可完整运行、现场演示并具备清晰业务闭环的网页 Demo  
**核心输入：** 500 字以内的中文短文本帖子及简版发布者画像  
**核心输出：** 模拟评论区、沟通风险诊断、修改方向、保持人设的改写文本及改写前后的反事实对比。

---

## 二、选题背景

个人创作者在发布社交媒体短帖、个人观点或公开说明前，往往只能从“自己想表达什么”出发检查文本，却难以同时考虑不同受众的阅读深度、初始信任、情绪敏感度以及评论区热门观点对后续讨论的影响。

一条原意清晰的帖子，可能因为模糊指代、限定条件缺失、语气过强或上下文不足，被解读为停更、辩解、攻击群体或回避责任。某些解读还可能因获得更多点赞和回复而成为评论区的主导叙事。

例如，一位创作者发布：

> 最近会适当降低更新频率，希望大家理解。

发布者原本只是想表达“短期减少更新”，但不同受众可能产生完全不同的理解：

- 核心老粉可能认为创作者需要休息；
- 普通关注者会追问具体更新频率；
- 大量潜水用户虽然不发言，但可能给“准备停更”的评论点赞，使这一解读成为评论区的主导叙事。

现有内容辅助工具多集中于文案润色、敏感词检测、情绪分析和发布后舆情监测，较少在发布前完成 **“受众构造—多轮互动模拟—风险归因—人设保持改写—同受众重模拟”的完整闭环**。因此，本项目拟构建一个轻量、可解释的多 Agent 发布前沟通风险压力测系统。

---

## 三、目标用户与使用边界

本项目主要面向：

- 普通社交媒体用户；
- 个人内容创作者；
- 知识、生活、娱乐或观点类博主；
- 需要发布个人公开说明的用户。

第一版拟设定以下边界：

- 不判断用户观点是否正确；
- 不进行事实核查；
- 不读取链接、图片和视频；
- 不将模拟结果表述为真实舆情概率或确定性预测。

---

## 四、核心使用场景

### 4.1 个人观点与日常表达

包括社会现象看法、行业事件评价、生活经验分享、作品发布及对某类行为的回应。系统将关注限定条件丢失、观点被过度概括、某句话被单独截取和讨论跑题等风险。

### 4.2 个人道歉或公开回应

包括对争议事件、不当言论或个人行为的回应。系统将关注责任表达不清、解释被读成辩解、语气与人设不符以及受众关注点错位等问题。

系统拟自动识别内容类型，不要求用户在输入前手动选择。

---

## 五、项目核心定位

> CommentLab 拟定位为面向个人创作者的发布前沟通鲁棒性测试 Agent。系统通过不同受众的多轮阅读、评论、回复和点赞模拟，识别误解、负面情绪、争吵对立和话题跑偏风险，生成保持发布者人设的改写文本，并在相同受众与随机配置下进行反事实重模拟。

系统不拟回答“真实网友一定会怎样评论”，而是希望帮助用户发现：

- 哪些表达存在解释空间；
- 哪些误读可能因热度和回复继续放大；
- 如何在不改变核心观点和个人语气的前提下降低沟通风险。

## 六、相关研究与项目差异

已有研究已经证明，大语言模型可以用于构建虚拟社区和社会互动模拟。

Social Simulacra 根据社区目标、规则和成员画像生成帖子、回复及反社会行为，并支持通过改变社区设计观察不同的“what-if”结果。该工作主要服务于社交计算系统和社区设计原型验证。

S³ 使用大语言模型 Agent 模拟社交网络中的情绪、态度和互动行为，并观察信息、情绪和观点在群体中的传播。

OASIS 提供了一个开源、可扩展的社交媒体仿真环境，支持发帖、评论、关注、推荐和动态社交网络等机制，并可研究信息传播、群体极化和从众效应。

这些研究说明，多 Agent 社交模拟具有可行性，但它们更多关注：

- 大规模社会网络现象；
- 平台机制；
- 群体极化；
- 信息传播；
- 社区规则和治理；
- 宏观社会实验。

本项目不拟复刻完整社交平台，而是聚焦于一条内容发布前的轻量压力测试，回答一个更具体的使用问题：

> 普通创作者在一条内容尚未发布时，如何发现其中容易被误读的表述，并验证修改后的文本是否更稳健

预期差异包括：

1. **从发布前而非发布后介入**；
2. **面向普通个人和个人创作者**；
3. **将风险归因到具体文本片段**；
4. **追踪从原文到误读、再到热度放大的误解链**；
5. **加入发布者画像与人设保持改写**；
6. **使用同一批受众进行反事实重模拟**。

## 七、拟定需求与约束

| 项目 | 拟定方案 |
| --- | --- |
| 内容类型 | 平台无关的中文短帖和个人公开说明 |
| 文本长度 | 不超过 500 个字符 |
| 发布者画像 | 身份、内容领域、粉丝规模、表达风格、受众关系 |
| 受众规划 | 系统自动规划，用户可调整大类比例和 Persona 权重 |
| 模拟轮数 | 默认 3 轮 |
| 潜水用户 | 使用 Python 规则模拟点赞，不调用模型 |
| 核心风险 | 误解、负面情绪、争吵对立、话题跑偏 |
| 人设保持 | 改写时不撤回观点、不添加未知事实、不统一改成官方声明 |
| 反事实对比 | 固定 Persona、比例、随机种子、轮数和热度规则 |
| 外部语料 | 不作为第一版运行前置，仅留作后续评测和语言自然度增强 |
| 可视化 | 优先实现结果对照与树状评论，不做大型网络图 |

## 八、系统功能设计

### 8.1 内容与发布者画像输入

用户输入中文短帖，并填写身份、领域、粉丝规模、表达风格和与受众的关系。

### 8.2 内容语义与沟通风险初步分析

系统拟分析：

- 核心表达和主要观点；
- 内容类型与语气；
- 文本明确表达的信息；
- 模糊指代、缺失信息和情绪化表达；
- 可能被截取或过度概括的片段；
- 与发布者人设存在张力的语气。

这些结果将作为后续 agent 评论的语义依据。

### 8.3 受众 Persona 规划

拟设置核心老粉、普通关注者、路人、标题阅读者、理性追问者、专业纠错者、动机怀疑者、情绪共鸣者、争议放大者和潜水点赞者等类型。

每个发言型 Persona 拟包含阅读深度、对发布者的信任、情绪敏感度、争议倾向和影响力等参数。模型只负责规划受众构成，底层参数由本地模板约束，以降低输出不稳定性。

### 8.4 多轮评论与多分支互动模拟

预计默认激活 8～10 个发言 Persona，配置约 50 个规则化潜水用户，执行 3 轮互动：

1. **第一轮：** 多个 Persona 根据原帖形成首轮反应；
2. **第二轮：** 根据可见热评发起新顶级评论或加入不同讨论分支；
3. **第三轮：** 继续观察误解扩散、冲突升级、话题跑偏或讨论收束。

评论 Agent 拟先确定关注的原文或父评论片段，再选择支持、个人联想、第一印象、质疑、调侃、动机猜测或争议化概括等反应类型，最后生成评论。这一设计用于提升评论对当前帖子的针对性和 Persona 区分度。

### 8.5 热度与潜水用户规则

热度拟综合点赞、回复数、情绪强度、争议性与时间衰减：

```python
hot_score = (
    likes
    + 1.5 * reply_count
    + 2.0 * emotion_intensity
    + 1.5 * controversy
) / age_factor
```

潜水用户不调用模型，而是通过立场匹配、已有热度、情绪强度、语句简短程度、传播性和 Persona 影响力模拟点赞。

### 8.6 风险诊断与改写

系统拟识别四类风险：

- 内容误解；
- 负面情绪；
- 争吵和阵营对立；
- 话题跑偏。

诊断结果拟包含高风险原文片段、风险原因、误解链、修改方向和综合结论。改写 Agent 将在不改变核心观点、不增加未知事实、保留个人语气的前提下，优先修复高风险表达。

### 8.7 反事实二次模拟

改写后拟使用同一批 Persona、比例、随机种子、潜水用户数、模拟轮数和热度规则重新运行。预计比较综合风险、误解评论比例、负面评论比例、对立回复数和跑题比例。

---

## 九、风险评估方案

四类风险拟采用 1～5 的内部分值，权重初步设定为：

```python
simulation_score = (
    misunderstanding_risk * 0.40
    + negative_emotion_risk * 0.30
    + conflict_risk * 0.20
    + off_topic_risk * 0.10
)
```

文本分析和模拟结果的综合权重初步设定为：

```python
final_score = text_analysis_score * 0.25 + simulation_score * 0.75
```

风险等级划分：

```python
if final_score < 2.0:
    level = "低"
elif final_score < 3.5:
    level = "中"
else:
    level = "高"
```

上述权重属于项目初期的可解释启发式设定，后续拟通过人工标注案例和小规模用户评估进行校准。界面只向用户显示低、中、高及其解释，不将分数表述为真实发生概率。

## 十、Agent 与工具架构

1. **内容分析 Agent：** 提取核心表达、内容类型、语气、明确信息、合理分歧、模糊表达和潜在误读。
2. **受众规划 Agent：** 根据帖子和发布者画像规划 Persona 类型与比例。
3. **评论生成 Agent：** 依据帖子语义、Persona 属性和可见评论，批量生成评论、回复、点赞或沉默行为。
4. **热度模拟工具：** 使用 Python 完成潜水点赞、回复计数、热度更新和排序。
5. **风险诊断 Agent：** 结合原文和模拟结果定位风险片段、误解链和修改方向。
6. **人设保持改写 Agent：** 在保留核心观点和个人风格的条件下生成改写版本。
7. **对比评估 Agent：** 综合改写前后模拟指标和风险报告形成对比结论。

拟采用的固定链式流程为：

```text
内容分析
→ 受众规划
→ 用户确认与调整
→ 原文三轮模拟
→ 风险诊断
→ 人设保持改写
→ 改写后三轮模拟
→ 反事实对比
```

## 十一、核心数据设计

拟使用 Pydantic 约束结构化输入输出。除基本评论字段外，计划保存评论的语义依据和反应类型：

```python
class Comment(BaseModel):
    comment_id: str
    persona_id: str
    persona_label: str
    parent_id: str | None
    round_no: int
    text: str
    stance: str
    emotion: str
    emotion_intensity: float
    controversy: float
    misunderstanding: bool
    off_topic: bool
    evidence_span: str
    reaction_type: str
    likes: int
    reply_count: int
    hot_score: float
```

其中：

- `evidence_span` 用于记录评论实际关注的原文或父评论片段；
- `reaction_type` 用于记录支持、个人联想、第一印象、质疑、调侃或争议化概括等反应方式。

这两类字段将用于检查评论是否真正回应当前内容，并支持后续的评论筛选和风险解释。

## 十二、系统界面设计

预计使用 Streamlit 构建四阶段页面：

### 12.1 输入页

- 帖子文本与字数提示；
- 发布者身份、领域、粉丝规模、表达风格和受众关系。

### 12.2 受众确认页

- 内容初步分析；
- 受众大类比例；

### 12.3 原文结果页

- 左侧展示风险分析、高风险片段、修改方向和误解链；
- 右侧展示具有父子层级的树状模拟评论区。

### 12.4 改写对比页

- 展示原文、改写结果、风险判断、高风险片段、修改方向、误解链和两侧评论区；

系统还拟提供历史结果加载、逐级返回和阶段切换后自动回到页面顶部等基本交互。

## 十三、评测方案

### 13.1 功能完整性

检查从帖子输入、受众确认、原文模拟、风险诊断、改写到反事实对比的完整流程是否可运行。

### 13.2 结构化输出稳定性

检查字段缺失、JSON 解析、无效回复目标和评论长度。

### 13.3 评论仿真性

检查：

- 自定义帖子是否能形成多个顶级讨论分支；
- 评论是否存在重复和高频机械句式；
- 不同 Persona 是否表现出足够多样的反应类型。

### 13.4 风险识别与改写评估

拟通过人工标注比较风险点覆盖率，并从原语气保留、观点保持、官方化程度和作者一致性四个方面对改写文本进行评估。

## 十五、项目风险与局限

1. 多 Agent 模拟不能精准预测真实舆论；
2. Persona 可能过度脸谱化，评论也可能语言同质化；
3. 风险等级是内部启发式指标，不是客观概率；
4. 模型 API 延迟、超时和结构化输出失败可能影响演示；
5. 改写后模拟风险降低，不代表真实发布一定不会产生争议。

---

## 十六、项目创新点

1. 将多 Agent 社交模拟应用于内容发布前的轻量沟通风险测试；
2. 从原文片段、受众反应到热度放大追踪误解链；
3. 使用多顶级评论和多回复分支模拟评论区中的并行讨论；
4. 使用同一批受众和随机配置比较改写前后的模拟结果；
