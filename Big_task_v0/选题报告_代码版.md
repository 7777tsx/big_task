# 多 Agent 社交媒体内容发布风险模拟系统选题报告

## 一、项目基本信息

**项目暂定名称：** CommentLab  
**中文名称：** 面向个人创作者的多 Agent 社交媒体内容发布风险模拟系统  
**项目类型：** Agent 系统课程作业  
**开发形式：** 两人小组  
**目标形态：** 能够完整运行、现场演示、形成清晰闭环的网页 Demo  
**主要技术栈：** Python + Streamlit + LangChain + OpenAI-compatible API + SQLite  
**核心输入：** 500 字以内的中文短文本帖子及简版发布者画像  
**核心输出：** 模拟评论区、风险诊断、修改方向、保持人设的改写文本，以及修改前后的二次模拟对比

---

## 二、选题背景

个人创作者在发布社交媒体帖子、个人观点或公开说明前，往往难以准确判断受众会如何理解自己的表达。发布者关注的通常只是“我想表达什么”，但受众实际接收到的内容还会受到以下因素影响：

- 发布者身份及既有人设；
- 受众与发布者之间的关系；
- 受众是否完整阅读内容；
- 受众对发布者的初始信任；
- 模糊表述和缺失信息；
- 评论区中已经形成的热门观点；
- 用户跟随高赞评论的倾向；
- 情绪化表达带来的二次放大；
- 其他用户的回复、反驳和调侃。

例如，一位创作者发布：

> 最近会适当降低更新频率，希望大家理解。

发布者原本只是想表达“短期减少更新”，但不同受众可能产生完全不同的理解：

- 核心老粉可能认为创作者需要休息；
- 普通关注者会追问具体更新频率；
- 只看标题的用户可能理解为即将停更；
- 怀疑型用户可能猜测账号数据下降；
- 玩梗型用户可能将其概括为“退网倒计时”；
- 大量潜水用户虽然不发言，但可能给“准备停更”的评论点赞，使这一解读成为评论区的主导叙事。

当前常见的内容辅助工具多集中于文案润色、情绪分析、敏感词检测和发布后的舆情监测。它们通常能回答“这段文案是否通顺”或“当前评论区情绪如何”，但较少完整模拟以下闭环：

> 内容发布前，构造不同类型的受众，模拟评论区的多轮互动，追踪误解如何产生和扩散，定位具体风险句，生成保持发布者人设的改写文本，并通过相同受众的二次模拟验证修改是否有效。

因此，本项目拟构建一个轻量、可解释、可运行的多 Agent 内容发布风险模拟系统，为普通个人和内容创作者提供发布前的沟通风险测试。

本项目主要服务于：

- 普通社交媒体用户；
- 个人内容创作者；
- 知识博主；
- 生活类、娱乐类或观点类创作者；

第一版不面向政府、学校行政部门或大型企业公关团队，不做复杂的组织级危机公关和跨平台舆情预测。

## 四、核心使用场景

系统第一版重点面向两类内容，但不要求用户手动选择帖子类型。

### 4.1 个人观点类内容

例如：

- 对某个社会现象的个人看法；
- 对行业事件的评价；
- 对争议话题的态度表达；
- 对粉丝或其他创作者行为的回应。

此类内容容易产生的风险包括：

- 立场被过度概括；
- 限定条件被忽略；
- 某一句话被单独截取；
- 观点被解释为对某一群体的攻击；
- 评论区形成阵营对立；
- 讨论偏离原始主题。

### 4.2 个人道歉或公开回应

例如：

- 对错误言论进行说明；
- 对争议事件公开回应；
- 向粉丝或合作对象道歉；
- 解释此前行为的原因。

此类内容容易产生的风险包括：

- 责任表达不清；
- 解释被理解为辩解；
- 语气与既有人设不一致；
- 道歉重点与受众关注点错位；
- 过多强调自身感受；
- 引出新的争议点。

系统不会判断“观点是否正确”或“道歉是否合格”，而是统一诊断内容发布后的潜在沟通风险。

## 五、项目核心定位

本项目定位为：

> 面向个人创作者的发布前沟通鲁棒性测试 Agent。系统通过不同受众 Agent 对帖子进行多轮阅读、评论、回复和点赞模拟，识别内容被误解、引发负面情绪、形成争吵以及话题跑偏的潜在路径，并在保持发布者人设的前提下给出改写方案，再通过反事实重模拟比较修改前后的风险变化。

项目不承诺精准预测真实评论区，不输出“真实网友一定会这样评论”的结论，而是将模拟评论作为风险分析的证据界面，用于回答：

- 哪些表达存在沟通风险；
- 哪类受众最容易产生不同理解；
- 哪些理解可能因点赞和回复逐渐成为主导叙事；
- 哪些信息缺失导致受众自行补充动机；
- 如何修改能够降低误解和冲突风险。

## 六、相关研究与研究空白

### 6.1 现有多 Agent 社交模拟研究

已有研究已经证明，大语言模型可以用于构建虚拟社区和社会互动模拟。

Social Simulacra 根据社区目标、规则和成员画像生成帖子、回复及反社会行为，并支持通过改变社区设计观察不同的“what-if”结果。该工作主要服务于社交计算系统和社区设计原型验证。

S³ 使用大语言模型 Agent 模拟社交网络中的情绪、态度和互动行为，并观察信息、情绪和观点在群体中的传播。

OASIS 提供了一个开源、可扩展的社交媒体仿真环境，支持发帖、评论、关注、推荐和动态社交网络等机制，并可研究信息传播、群体极化和从众效应。

这些研究说明，多 Agent 社交模拟具有可行性，但它们更多关注：

- 大规模社会网络现象；
- 平台机制；
- 群体极化；
- 信息传播；
- 社区规则和治理；
- 宏观社会实验。

### 6.2 本项目拟解决的空白

本项目不尝试重新实现大规模社交平台，而是聚焦一个更具体的使用问题：

> 普通创作者在一条内容尚未发布时，如何发现其中容易被误读的表述，并验证修改后的文本是否更稳健？

本项目的差异化重点包括：

1. **发布前而非发布后。**
2. **面向普通个人和创作者。**
3. **重点追踪误解链。**
4. **风险归因到具体文本。**
5. **加入发布者身份和人设。**
6. **保持人设进行改写。**
7. **使用同一批受众进行反事实重模拟。**

## 八、系统功能设计

### 8.1 发布者画像输入

用户填写以下五项：

```json
{
  "identity": "科技知识博主",
  "domain": "人工智能与数码",
  "follower_scale": "中小体量",
  "style": "理性、直接、偶尔幽默",
  "audience_relationship": "长期关注者较多，粉丝信任度较高"
}
```

### 8.2 帖子输入

用户粘贴不超过 500 个汉字的中文短帖。第一版不读取链接、图片、视频或历史账号内容，也不核查事实。

### 8.3 内容初步分析

系统分析：

- 内容核心表达；
- 主要观点；
- 涉及群体；
- 关键限定条件；
- 模糊词；
- 缺失信息；
- 可能引发情绪的表述；
- 容易被断章取义的句子；
- 与发布者人设不一致的语气；
- 潜在受众利益冲突。

示例：

```json
{
  "main_message": "创作者将暂时减少更新",
  "ambiguous_phrases": [
    {
      "text": "适当降低更新频率",
      "reason": "调整程度不明确"
    }
  ],
  "missing_information": [
    "具体频率",
    "持续时间",
    "调整原因"
  ],
  "possible_misreadings": [
    "创作者准备停更",
    "账号运营出现问题"
  ]
}
```

### 8.4 自动生成受众 Persona

| 大类   | Persona | 主要特征                |
| ---- | ------- | ------------------- |
| 粉丝群体 | 核心老粉    | 熟悉发布者，容忍度较高，对人设变化敏感 |
| 粉丝群体 | 普通关注者   | 关注是否影响自己，阅读较完整      |
| 路人群体 | 普通路人    | 不熟悉发布者，只根据当前帖子判断    |
| 路人群体 | 只看标题型   | 阅读深度低，容易丢失限定条件      |
| 理性群体 | 理性追问者   | 关注时间、范围、原因和具体数据     |
| 理性群体 | 专业纠错者   | 关注逻辑、术语和表达准确性       |
| 怀疑群体 | 动机怀疑者   | 容易将信息缺口解释为隐藏动机      |
| 怀疑群体 | 商业化敏感者  | 对广告、付费和商业合作敏感       |
| 情绪群体 | 情绪共鸣者   | 容易受语气和高赞评论影响        |
| 互动群体 | 玩梗调侃者   | 容易将某句话变成梗并带偏话题      |
| 互动群体 | 争议放大者   | 将疑问改写成更绝对的说法        |
| 沉默群体 | 潜水点赞者   | 不发言，但影响热度和排序        |

### 8.5 受众比例调整

用户可调整：

- 一级：粉丝、路人、理性、怀疑、情绪与互动等大类比例；
- 二级：核心老粉、只看标题型、动机怀疑者、理性追问者等具体 Persona 比例。

第一版不开放底层参数修改。

### 8.6 多轮评论模拟

默认配置：

- 8～10 个发言型 Persona；
- 约 50 个规则化潜水用户；
- 3 轮互动；
- 最终约 12～20 条评论。

#### 第一轮：初始反应

6～8 个 Agent 只看原帖，生成首轮评论。

#### 热度更新

Python 模拟潜水用户点赞并计算热度。

#### 第二轮：热点形成

3～5 个 Agent 根据高热评论回复、反驳、补充或跟随。

#### 第三轮：扩散或收束

2～4 个 Agent 继续互动，观察误解是否扩散、争论是否升级、话题是否偏移。

### 8.7 评论热度规则

```python
hot_score = (
    likes
    + 1.5 * replies
    + 2.0 * emotion_intensity
    + 1.5 * controversy
) / age_factor
```

潜水用户点赞概率受以下因素影响：

- 立场匹配；
- 已有热度；
- 情绪强度；
- 表达简短程度；
- 传播性；
- Persona 影响力。

### 8.8 评论结果展示

全部模拟评论按热度排序，展示：

- Persona 标签；
- 评论内容；
- 模拟点赞数；
- 回复关系。

不在评论旁展示风险标签。

## 九、风险分析体系

### 9.1 内容误解风险

检测：

- 范围扩大；
- 限定条件丢失；
- 不确定变成确定；
- 个别变成普遍；
- 动机脑补；
- 前提忽略；
- 时间扩大；
- 反向理解。

### 9.2 负面情绪风险

检测：

- 愤怒；
- 失望；
- 焦虑；
- 被欺骗感；
- 不被尊重感；
- 嘲讽；
- 不信任。

### 9.3 争吵和阵营对立风险

检测：

- 支持者与反对者连续互相回复；
- 从内容争论转为身份攻击；
- 群体概括；
- 某一评论引发连续反驳；
- 讨论对象从帖子转向评论者。

### 9.4 话题跑偏风险

检测：

- 转向发布者过去行为；
- 转向人格评价；
- 玩梗压过原始议题；
- 扩展到无关群体；
- 高赞评论改变讨论中心。

## 十、综合风险等级

大模型先给出四项内部评分：

```json
{
  "misunderstanding_risk": 4,
  "negative_emotion_risk": 3,
  "conflict_risk": 2,
  "off_topic_risk": 2
}
```

固定权重：

```python
total = (
    misunderstanding_risk * 0.40
    + negative_emotion_risk * 0.30
    + conflict_risk * 0.20
    + off_topic_risk * 0.10
)
```

综合原文分析和模拟结果：

```python
final_score = (
    text_analysis_score * 0.25
    + simulation_score * 0.75
)
```

风险等级：

```python
if final_score < 2.0:
    level = "低"
elif final_score < 3.5:
    level = "中"
else:
    level = "高"
```

最终只向用户显示低、中、高及其解释，不显示具体百分制。

## 十一、风险诊断输出

系统输出：

1. 高风险原文片段；
2. 风险形成原因；
3. 可能出现的误解链；
4. 具体修改方向；
5. 保持人设的改写文本。

示例：

> 原文片段：“有些人真的应该反思一下。”

风险：

- “有些人”指代不明确；
- 多个群体可能认为自己受到攻击；
- 容易触发“到底在说谁”的猜测；
- 适合被脱离上下文截图传播。

修改方向：

1. 明确针对的是某类行为，而不是某类人；
2. 将人格评价改为行为评价；
3. 补充具体背景；
4. 减少绝对化表达；
5. 保留发布者直接的语气。

---

## 十二、保持人设的改写

要求：

- 不改变核心观点；
- 不替用户撤回立场；
- 不统一改成官方声明；
- 保留原语气和用词习惯；
- 优先修复高风险句；
- 补充必要信息；
- 不添加用户未提供的事实；
- 长度尽量接近原文。

---

## 十三、反事实二次模拟

改写后使用同一批受众 Agent 重跑模拟。

固定：

- Persona 类型与比例；
- 底层属性；
- 潜水用户数量；
- 模拟轮数；
- 热度算法；
- 随机种子；
- 每轮激活人数。

比较：

- 综合风险等级；
- 误解评论比例；
- 负面情绪比例；
- 对立性回复数量；
- 跑题比例；
- 主要误解是否仍出现；
- 高频追问是否变化。

## 十四、Agent 架构

### 14.1 内容分析 Agent

负责提取核心信息、模糊表达、缺失信息、潜在误解和人设冲突。

### 14.2 受众规划 Agent

负责根据帖子和发布者画像生成 Persona 构成及比例。

### 14.3 评论生成 Agent

负责按 Persona 生成发言、回复或沉默行为。为节约 token，同一轮多个 Persona 合并为一次模型调用。

### 14.4 热度模拟工具

Python 实现，不使用大模型。负责潜水用户点赞、回复计数、热度计算、排序和可见评论选择。

### 14.5 风险诊断 Agent

负责综合原文和评论，识别四类风险、定位风险句、提取误解链和生成修改方向。

### 14.6 人设保持改写 Agent

负责在不改变核心观点的前提下进行改写。

### 14.7 对比评估 Agent

负责比较原文和改写版本的模拟结果。

## 十五、LangChain 实现方案

### 15.1 结构化输出

使用 Pydantic 模型约束输出：

```python
from pydantic import BaseModel, Field
from typing import Literal

class CommentAction(BaseModel):
    persona_id: str
    action: Literal["comment", "reply", "like", "ignore"]
    target_comment_id: str | None = None
    text: str | None = None
    stance: Literal["support", "oppose", "question", "neutral"]
    emotion: str
    emotion_intensity: float = Field(ge=0, le=1)
```

调用：

```python
structured_model = llm.with_structured_output(CommentBatch)
result = structured_model.invoke(prompt)
```

### 15.2 固定链式流程

```text
内容分析
→ 受众生成
→ 用户调整
→ 原文三轮模拟
→ 风险诊断
→ 改写
→ 改写后三轮模拟
→ 前后对比
```

第一版使用 LangChain Runnable 或普通链式调用，不需要复杂的无限循环。

## 十六、Token 控制

推荐模型调用量：

| 阶段      | 调用次数 |
| ------- | ----:|
| 内容分析    | 1    |
| 受众生成    | 1    |
| 原文三轮模拟  | 3    |
| 原文风险分析  | 1    |
| 改写      | 1    |
| 改写后三轮模拟 | 3    |
| 前后对比    | 1    |
| 合计      | 约 11 |

控制方式：

- 一次请求生成多个 Persona 的行为；
- 每轮只传 3～5 条可见评论；
- Persona 描述参数化、短文本化；
- 评论长度限制为 10～60 字；
- 潜水用户全部用 Python 规则；
- 对相同输入进行缓存；
- 准备离线演示案例。

## 二十、数据来源

### 20.1 CHECKED

公开中文微博数据，包含微博、评论、转发、点赞及传播信息。

下载：

```bash
git clone https://github.com/cyang03/CHECKED.git
```

用途：

- 中文评论风格；
- 原帖与评论关系；
- 评论、转发和点赞结构；
- few-shot 示例。

局限：主题集中在新冠疫情。

### 20.2 ResEmo

包含 3813 条微博帖子和 68,781 条评论，适合响应情绪和情绪原因分析。公开论文可获取，但目前未确认稳定公开下载入口，可联系作者申请，不能作为项目运行前置条件。

### 20.3 GoEmotions

约 5.8 万条英文 Reddit 评论，标注 27 类细粒度情绪和中性。适合建立情绪分类体系，不适合直接模拟中文语言风格。

### 20.4 Mastodon 公开线程

官方 API 可通过状态上下文获取父级和子级状态，用于参考回复树和热点评论结构。建议离线保存少量去标识化线程，不让系统依赖实时接口。

### 20.5 自建中文评测集

建议整理 20～30 个案例，每个案例 20～50 条去标识化评论。

标注：

- 立场；
- 情绪；
- 是否误解；
- 是否跑题；
- 议题；
- 回复对象。

## 二十一、评测方案

### 21.1 功能完整性

验证完整闭环是否成功运行。

### 21.2 结构化输出稳定性

检测字段缺失、JSON 解析、重复评论、无效回复目标等问题。

### 21.3 Persona 区分度

人工判断不同 Persona 是否真的表现出不同阅读和表达方式。

### 21.4 风险点覆盖率

```text
风险点覆盖率 =
系统识别出的真实主要争议点
÷
真实主要争议点总数
```

### 21.5 改写有效性

比较原文与改写：

- 模拟内指标；
- 人工理解结果；
- 负面解读；
- 追问变化；
- 人设保持程度。

### 21.6 人设保持评价

人工按 1～5 分评估：

- 是否保留原语气；
- 是否改变观点；
- 是否过度官方；
- 是否仍像同一个人写的。

## 二十六、局限与风险

1. 模拟不能精准预测真实舆论；
2. LLM Agent 可能语言同质化；
3. 压力测试可能生成较多负面内容；
4. 风险等级不是客观概率；
5. 自建数据必须去标识化；
6. 第一版不做事实核查；
7. 修改后风险降低不代表内容一定不会引发争议；
8. 模型输出可能受随机性和 API 状态影响。
